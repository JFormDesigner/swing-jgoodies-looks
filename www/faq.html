What are the main benefits of the Looks?

The JGoodies Looks improve the readability, legibility, consistency and in turn the usability of your Java applications and applets. And many people have reviewed the JGoodies L&Fs as elegant. Also, the JGoodies Looks bring bug fixes from recent Java versions to older Java versions.

The JGoodies Windows Look&Feel focuses on a precise emulation on Windows 95/98/NT/ME/2000/2003/XP/Vista in the following areas: menus, icons, colors, borders, fonts, font sizes, insets, and widget dimensions. It honors the screen resolution (96dpi vs. 120 dpi) to adjust sizes, insets, and widget dimensions. And it honors the desktop font size setting in western Windows environments.

Plastic, Plastic3D, and PlasticXP are elegant multi-platform Look&Feels that look good on all Windows platforms, including XP. The Plastic themes have been tweaked for a harmonic integration with the majority of desktops so that your applications won't alienate a Windows user.

The JGoodies Looks simplify the support for different Java versions, operating systems, and Look&Feels. Each look is consistent over the Java versions 1.4.1x, 1.4.2x, 1.5, and 1.6 betas - to the possible extent w. r. t. to the underlying Swing implementation. And all JGoodies L&Fs use similar component sizes to minimize the layout differences when switching the Look&Feel.

Can I use the Looks in a commercial product?

You can use, copy and redistribute the JGoodies Looks in source and binary form - even for commercial use - if you comply with the very relaxed terms of the BSD open source license. The BSD license has been chosen, because it allows many uses, and the few license terms are short and human readable, even if you're not a lawyer.

How to get started?

First, open and read the README.html and learn about the available documentation: the RELEASE-NOTES, Quick Start, User's Guide, Tips & Tricks, JavaDocs. Then download and run the JGoodies Looks Demo. Follow the instructions in the User's Guide how to use a JGoodies Look&Feel, how to configure it, and how to use the optional features. You may also study the presentation "First Aid for Swing" at the JGoodies articles page.

Where do I get support?

Support is available only in the user mailing list users@looks.java.net. Before you post, check this FAQ, the relevant JavaDocs, the User's Guide (available from the README.html), and the accompanying demo sources. You can find and track the progress of Looks issues in the issue tracker. As a project observer you can log defects, request features and suggest enhancements.

The user mailing list helps us reduce the costs for managing this freeware project. In the mailing list all project users can jump in to assist you, if you face a problem with the library or API. And other users can benefit from your questions and our answers. Personal mail to JGoodies.com bypasses the other project developers, and it often leads to unnecessary extra work; it slows down the development process, and it costs time and money on our side.

If you have downloaded the library at no charge, I ask you to register with the user mailing list before you post, and post only with the registered account. Otherwise your messages won't show up automatically, and I need to manually accept every message.

Priority support is available for licensees of the commercial JGoodies Smart Client. These can contact us by phone or direct questions to support at JGoodies dot com.

If you have general questions about Swing and Java look&feels, please post to the Java Desktop forums at www.java.net. That's a great place to share information with the Java desktop community. And we frequently check this forum and participate in discussions.

How to report a problem?

Before you report a problem, check this FAQ, the relevant JavaDocs, the User's Guide (available from the README.html), and the accompanying demo sources. Important: please check if the problem is in the Java core or in the JGoodies Looks. Basically bugs in the Java core shall be reported to Sun, bugs in the JGoodies to the JGoodies project.

We don't accept reports for bugs that are in the Java core. There are many issues with the Sun L&fs in each individual Java version. And there are even more issues in all Java versions that run the JGoodies Looks. But we can't do more than Sun's Swing team does for a single Java version. Most of the Java core issues "shine through" to the JGoodies L&fs. So if you can reproduce a Plastic issue with Sun's Metal L&F, then report it to Sun. If you can reproduce a JGoodies Windows issue with Sun's Windows L&F, report it to Sun.

If you think you've found a bug in the JGoodies Looks, and again, not in the Java core but in the JGoodies Looks, you may check back in the issue tracker or may ask in the user mailing list, if this issue is already known.

If you report a bug, please provide at least the following information: JRE vendor, JRE version, JRE build, OS, OS version, Looks version, Look&feel used, and whether the problem can be reproduced with the JGoodies Looks Demo or not. For example: Sun Java 1.5.0_06, Windows XP, Looks 2.0, PlasticXP, can be reproduced with the JGoodies Looks Demo. Please point us to the Looks Demo, if it shows a problem. If you cannot reproduce the problem with the Looks Demo, please describe the steps to produce the problem. Often a screenshot makes it easier to understand a problem, or a request for an enhancement.

What are the differences between the Sun and JGoodies Windows L&Fs?

The JGoodies Looks 2.0.1 uses more consistent fonts, font baseline positions, component sizes, and insets than the Sun Windows L&f. On western Windows the JGoodies Windows L&f honors the desktop font size setting Normal/Large/Extra Large. And the JGoodies Looks brings bug fixes from recent Java versions to older Java versions.

The Sun Windows L&F provides a better live update of colors and fonts. If the user changes the desktop settings these changes shall be reflected in Swing immediately. This works well with the Sun Windows L&f. The color and fonts corrected by the JGoodies Windows L&f may change immediately or may require an application restart. Also, the JGoodies Windows L&F may lag slightly behind the most recent Sun Windows L&F bug fixes. This is because the Sun or JGoodies code may include a bug fix earlier than the other. Most Sun bug fixes are inherited by the JGoodies implementation, but not every fix. For example when Sun introduced subpixel anti-aliasing in an early Java 6 preview, the JGoodies implementation supported this feature with a delay of 2 months. But we always try to catch up with the core before a new major Java version is available.

We are interested in moving the fixes and changes made in the JGoodies Looks to the Java core. This is the best way to help the Swing community get a great Windows appearance. We were invited to join the Mustang effort lately. But more JGoodies improvements shall be moved to the Java 7 core. Hopefully this will include the consistent micro design. A proposal for an approach to move this to the core is tracked under issue #116.

What are the differences between WinLAF and the JGoodies Windows L&F?

As of the Looks 2.0 and WinLAF 0.5.1, the JGoodies Windows l&f provides a more precise micro design for 96dpi and 120dpi which leads to better readability and legibility, especially if used with a precise layout system such as the JGoodies Forms. The JGoodies Windows l&f uses consistent component sizes and insets, and it follows the "Windows Layout Specifications and Guidelines" and the "Windows Vista UX Guidelines" 0.8. Unlike WinLAF, the JGoodies L&f can be uninstalled properly at runtime enabling a hot L&f switch. Last but not least, the Looks 2.0 support Java 6 and Windows Vista.

Currently both the winlaf and JGoodies l&f fix bugs in the Sun Windows l&f that are not addressed by the other project. In 2005 WinLAF and JGoodies started to join their efforts in making the Windows look&feel look and feel like Windows. Several fixes could be moved to the Sun Windows l&f in Java 6.

How to use a JGoodies Look&Feel?

try {
   UIManager.setLookAndFeel(new PlasticXPLookAndFeel());
} catch (Exception e) {}
See also the section "Look & Theme" in the Looks' User's Guide.

How to set a Plastic color theme?

PlasticLookAndFeel.setPlasticTheme(new DesertBlue());
try {
   UIManager.setLookAndFeel(new Plastic3DLookAndFeel());
} catch (Exception e) {}
See also the section "Look & Theme" in the Looks' User's Guide.

How to remove the JTabbedPane content border?

JTabbedPane tabbedPane = new JTabbedPane();
tabbedPane.putClientProperty(
    Options.NO_CONTENT_BORDER_KEY, 
    Boolean.TRUE);

ClassNotFoundExceptions when launched with Java Web Start

If you use a third party l&f in a network launchable environment such as Java Web Start, you must indicate where to find the l&f classes: 

UIManager.put(
    "ClassLoader", 
    LookUtils.class.getClassLoader());

The Plastic color theme I set is ignored.

To make use of a Plastic color theme, you must set the theme before you set the Plastic look&feel in the UIManager. See also the section "Look & Theme" in the Looks' User's Guide.

Why not put the Looks JAR into the extension directory?

First off, this issue is not specific to the JGoodies Looks, it affects all Java libraries.

If you have full control over the applications running in a Java environment, you can put a single library version per library into the extension path. If you have no control over the applications running in your Java environment, use the user classpath only! Many runtime environments out there are shared by multiple applications, and so one has no control over the classes loaded or required by these applications. In this case, do not put a library version into the extension classpath.

Here's the problem: let's say you have a library class C with method C#m1. In a second version of the same class (C') you add #m2. If you put C into the extension path, and have an application that includes C' in its JAR, the app will load C, not C'. And the application may fail with a NoSuchMethodException or other exception. If the implementation of #m1 differs in C and C', the application will load C and execute C#m1 and won't benefit from changes to #m1.

Even if you can use the extension directory, you may consider extending the user classpath, for example by referencing all required libraries in a main JAR's manifest. Using the user classpath makes it simpler to move an application to other Java environments that you don't control, for example to provide a Web start version.
